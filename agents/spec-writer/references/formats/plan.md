# plan.md フォーマット定義

機能追加の全情報を1ファイルに統合した設計・実装計画ドキュメント。
要件定義、技術設計（API/DB/フロントエンド）、実装タスク、テスト方針を1ファイルに含む。

**出力先ルール**:
- `docs/{feature-name}/` は **カレントディレクトリ（Claude Code 起動ディレクトリ）直下** に作成する
- `{feature-name}` は **日本語のシンプルな名前**（例: `キャンペーン通知`）。パス区切り（`/`）を含めない

## ドキュメント構成

```
docs/{feature-name}/
├── plan.md              # 全情報を統合した設計・実装計画
└── project-context.md   # プロジェクトコンテキスト（規約、技術スタック、パターン）
```

## 記述ルール

- YAML frontmatter で `title`, `status`, `date` を管理する
- `status` の遷移: `planning` → `ready`（以降は変更しない。実装状態は progress.md で管理）
- 受入条件は「〜できること」「〜が表示されること」のように検証可能な形で書く
- スコープ外を明示することで実装の膨張を防ぐ
- シーケンス図は必須。メインユースケースを1つ以上図示
- バックエンド/DB/フロントエンドの各セクションには **具体的なコード例** を含める
- TypeScript interface でリクエスト/レスポンス型を定義
- エラーコードはマシンリーダブルなコード文字列を付与
- ER図は Mermaid erDiagram で可視化
- マイグレーションは SQL で記述
- Props型は TypeScript interface で定義
- ASCIIワイヤーフレームを含める（フロントエンドがある場合）
- コンポーネントツリーは Mermaid で可視化
- ファイルパスはプロジェクトルートからの相対パスで記載する
- 既存パターンへの参照は「既存の `useXxx` のパターンに倣う」形式で記述する

## セクション省略ルール

全ての機能が全セクションを必要とするわけではない:

| セクション | 省略条件 |
|----------|---------|
| バックエンド変更 | API変更がない場合 |
| DB変更 | DB変更がない場合 |
| フロントエンド変更 | UI変更がない場合 |

以下は **常に必須**:
- 概要、課題と背景、受入条件、スコープ
- データフロー
- 実装タスク、テスト方針

省略するセクションには「該当なし」と記載する。

## 分割された機能の場合

機能が Phase 分割された場合、概要セクションの後に以下を追加:

```markdown
## Phase 管理

| Phase | 機能名 | 仕様書 | 依存 | ステータス |
|-------|--------|--------|------|----------|
| 1 | {機能名A} | [docs/{name-a}/](../name-a/) | - | {ステータス} |
| 2 | {機能名B} | [docs/{name-b}/](../name-b/) | Phase 1 | {ステータス} |
```
