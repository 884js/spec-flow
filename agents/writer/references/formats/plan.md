# plan フォーマット定義

機能追加の全情報を統合した設計・実装計画ドキュメント。

## セクション構成

以下のセクションを順に記述する。省略ルールに該当するセクションは **セクション自体を削除** する。

### 概要
- 背景・目的を簡潔に記述（1-3文）

### 関連プラン（関連する既存プランがない場合は省略）

- feature-name で参照する

| プラン | 関連 |
|--------|------|
| {feature-name} | 関連性の説明（自由記述） |

### ユーザーストーリー（ユーザー種別が単一かつ概要で十分な場合は省略）
- ユーザー種別ごとに `US-{ID}:` 形式で列挙
- 書式: `{ユーザー種別}として、{行動}したい。なぜなら{理由}だから。`

### 確認事項

仕様上の推測・仮定・未確定事項、およびデータ取得条件・除外ルール等のビジネスロジックをテーブル形式で列挙する。各項目にステータスを付与し、仕様の確信度を可視化する。

**仕様に明示されていないが、実装時に判断が必要になる条件を必ず確認事項として列挙すること。** 例: 対象データの範囲・条件、状態遷移の制約、エラー時の振る舞い、権限・アクセス制御。

| # | 項目 | 根拠 | ステータス |
|---|------|------|-----------|
| 1 | {確認事項} | {根拠・出典} | {✅確認済み / ⚠️要確認 / ❓要議論} |

- ✅確認済み: ユーザーまたはコードで確認が取れた事項
- ⚠️要確認: 推測に基づいているが、実装前に確認が望ましい事項
- ❓要議論: 複数の解釈があり、方針決定が必要な事項

### 追加検討事項（追加検討事項がない場合は省略）

仕様で触れられていないが、コードベース分析の結果として影響しそうな観点を列挙する。

| # | 観点 | 詳細 | 根拠 |
|---|------|------|------|
| 1 | {観点} | {詳細説明} | {コードのファイルパス等} |

### スコープ
- 「やること」「やらないこと」に分けて箇条書き

### 受入条件
- `- [ ] AC-N:` 形式で、検証可能な条件を列挙

### 非機能要件
- 該当なしの場合は「特になし」と記載

### データフロー
- Mermaid `sequenceDiagram` でメインユースケースを記述
- エラーフローがある場合は別図で記述

### バックエンド変更（API変更がない場合は省略）

#### API設計
- どんな操作を提供するか（CRUD等）
- 入力として何を受け取るか
- 出力として何を返すか
- 主要なエラーケースと条件

#### 対象ファイル
- `{新規/変更}: {ファイルパス} — {概要}` 形式

### DB変更（DB変更がない場合は省略）

#### データモデル

- 目的（何を管理するテーブルか）
- 関係（他テーブルとの関連）
- カラム表

| カラム | 型 | 制約 | デフォルト |
|--------|------|------|-----------|

#### 対象ファイル

### フロントエンド変更（UI変更がない場合は省略）

#### 画面・UI設計
- どんな画面/UIが必要か
- ユーザーが行える操作
- 表示するデータ

#### ワイヤーフレーム
- ASCII アートでレイアウトの意図を表現
- 画像が提供されている場合は Read で読み込み、画像のレイアウトをもとにワイヤーフレームを起こす
- 元画像のパスを記載: `元画像: {画像パス}`

例:
```
+----------------------------------+
| ヘッダー          [ログアウト]    |
+--------+-------------------------+
| サイド  | メインコンテンツ        |
| バー    |                        |
|         | +--------------------+ |
| - 項目1 | | カード              | |
| - 項目2 | | タイトル / 説明     | |
| - 項目3 | | [アクション]        | |
|         | +--------------------+ |
+--------+-------------------------+

元画像: /path/to/screenshot.png
```

## 記述ルール

- plan は **機能仕様書** であり、技術仕様書・実装ガイドではない
- ファイルパスはプロジェクトルートからの相対パスで記載する
- ソースコード（TypeScript、Prisma schema、SQL 等）は含めない。APIパス・カラム名・型名は表や箇条書きで記述する

## セクション省略ルール

| セクション | 省略条件 |
|----------|---------|
| 追加検討事項 | 追加検討事項がない場合 |
| 関連プラン | 関連する既存プランがない場合 |
| ユーザーストーリー | ユーザー種別が単一かつ概要で十分な場合 |
| バックエンド変更 | API変更がない場合 |
| DB変更 | DB変更がない場合 |
| フロントエンド変更 | UI変更がない場合 |
| 参考資料 | 参照すべき外部資料がない場合 |
| 参照更新箇所 | 波及的な変更が不要な場合 |

省略するセクションは **セクション自体を削除** する（「該当なし」と書かない）。

以下は **常に必須**: 概要、確認事項、受入条件、スコープ、データフロー、実装タスク、テスト方針

## 自己検証チェックリスト

生成中に以下を検証し、矛盾があれば生成中に修正すること:

| カテゴリ | チェック項目 |
|---------|------------|
| データフロー↔バックエンド | シーケンス図のAPI呼び出しがバックエンド変更に含まれているか |
| バックエンド↔DB | バックエンドのフィールドがDB変更のカラムと対応しているか |
| バックエンド↔フロントエンド | フロントエンドのデータがバックエンドのレスポンスと整合しているか |
| テスト網羅性 | 受入条件がテスト方針で全てカバーされているか |
| タスク網羅性 | 実装タスクが各セクションのファイルを全てカバーしているか |
| 依存関係 | タスク間の依存関係に循環がないか |
| 受入条件↔スコープ | 受入条件がスコープ「やること」の範囲内か |
| ユーザーストーリー↔受入条件 | 各USに対応するACが存在するか（USがある場合） |
| 確認事項の網羅性 | 仕様に明示されていないが実装時に判断が必要な条件が確認事項に挙がっているか |
| ドキュメント参照整合性 | 対象ファイルの削除・リネーム・役割変更が、他ドキュメント（README、他エージェント、他スキル）の参照と整合しているか |
| 用語・表記統一 | 新しい用語や表記の導入時、影響範囲内の全ドキュメントで統一されているか |
