# plan.md フォーマット定義

機能追加の全情報を1ファイルに統合した設計・実装計画ドキュメント。
要件定義、技術設計（API/DB/フロントエンド）、実装タスク、テスト方針を1ファイルに含む。

**出力先ルール**:
- `docs/plans/{feature-name}/` は **カレントディレクトリ直下** に作成する
- `{feature-name}` は **英語の kebab-case**（例: `campaign-notification`, `user-auth`）。パス区切り不可

## frontmatter

| フィールド | 必須 | 説明 |
|-----------|------|------|
| title | ○ | 機能タイトル（例: "feat: リマインダー機能"） |
| feature-name | ○ | kebab-case の機能名（例: "reminder"） |
| status | ○ | 常に `done` |
| created | ○ | 作成日（YYYY-MM-DD） |
| updated | ○ | 更新日（YYYY-MM-DD） |

## セクション構成

以下のセクションを順に記述する。省略ルールに該当するセクションは **セクション自体を削除** する（「該当なし」と書かない）。

### 概要
- ユーザーストーリー: 誰が・何を・なぜしたいのか（1-3文）

### 受入条件
- `- [ ] AC-N:` 形式で、検証可能な条件を列挙

### スコープ
- 「やること」「やらないこと」に分けて箇条書き

### 非機能要件
- 該当なしの場合は「特になし」と記載

### データフロー
- Mermaid `sequenceDiagram` でメインユースケースを記述
- エラーフローがある場合は別図で記述

### バックエンド変更（API変更がない場合は省略）

#### API設計
- どんな操作を提供するか（CRUD等）
- 入力として何を受け取るか
- 出力として何を返すか
- 主要なエラーケースと条件

#### 対象ファイル
- `{新規/変更}: {ファイルパス} — {概要}` 形式

### DB変更（DB変更がない場合は省略）

#### データモデル
テーブルごとに以下を記述:
- 目的（何を管理するテーブルか）
- 関係（他テーブルとの関連）
- カラム表

| カラム | 型 | 制約 | デフォルト |
|--------|------|------|-----------|

#### 対象ファイル

### フロントエンド変更（UI変更がない場合は省略）

#### 画面・UI設計
- どんな画面/UIが必要か
- ユーザーが行える操作
- 表示するデータ

#### ワイヤーフレーム
- ASCII アートでレイアウトの意図を表現

```
| - 項目3 | | [アクション]        | |
|         | +--------------------+ |
+--------+-------------------------+
```

- 影響範囲（影響するモジュール・機能）
- リスク（後方互換性、パフォーマンス、セキュリティ等）

## 記述ルール

- plan.md は **機能仕様書** であり、技術仕様書・実装ガイドではない
- ファイルパスはプロジェクトルートからの相対パスで記載する
- ソースコード（TypeScript、Prisma schema、SQL 等）は含めない。APIパス・カラム名・型名は表や箇条書きで記述する

## セクション省略ルール

| セクション | 省略条件 |
|----------|---------|
| バックエンド変更 | API変更がない場合 |
| DB変更 | DB変更がない場合 |
| フロントエンド変更 | UI変更がない場合 |

省略するセクションは **セクション自体を削除** する（「該当なし」と書かない）。

以下は **常に必須**: 概要、受入条件、スコープ、データフロー、実装タスク、テスト方針

## 自己検証チェックリスト

生成中に以下を検証し、矛盾があれば生成中に修正すること:

| カテゴリ | チェック項目 |
|---------|------------|
| データフロー↔バックエンド | シーケンス図のAPI呼び出しがバックエンド変更に含まれているか |
| バックエンド↔DB | バックエンドのフィールドがDB変更のカラムと対応しているか |
| バックエンド↔フロントエンド | フロントエンドのデータがバックエンドのレスポンスと整合しているか |
| テスト網羅性 | 受入条件がテスト方針で全てカバーされているか |
| タスク網羅性 | 実装タスクが各セクションのファイルを全てカバーしているか |
| 依存関係 | タスク間の依存関係に循環がないか |
| 受入条件↔スコープ | 受入条件がスコープ「やること」の範囲内か |
