# 不具合調査（Troubleshoot）

**原則: 原因が特定されるまでコードを修正しない。**

このスキルの目的は「直す」ことではなく「なぜ壊れているかを突き止める」こと。
推測に基づく修正は禁止。実際の実行フローを追い、事実に基づいて原因を特定する。
調査用の一時デバッグログの追加や検証スクリプトの作成は許可（修正ではなく調査手段）。

## ワークフロー

```
Step 0: コンテキスト判定（feature あり / なし）
Step 1: 症状の整理 + パターン判定
Step 2: 実行フローのトレース（パターンに応じた調査）
Step 3: 原因箇所の特定
Step 4: 修正方針の提示 + 調査ドキュメント出力
```

---

## Step 0: コンテキスト判定

このスキルは 2 つのモードで動作する:

- **feature モード**: `docs/plans/{feature-name}/` が存在する場合。調査ドキュメントを feature ディレクトリに保存し、plan.md や過去の調査と連携する
- **standalone モード**: feature ディレクトリがない場合。調査結果をユーザーに直接提示し、ドキュメント保存は任意

### 0-a. モード判定

$ARGUMENTS に feature-name が指定されている場合はそのディレクトリを使用。
指定がなければ以下で判定:

```
Glob docs/plans/**/plan.md
```

- plan.md が見つかった場合 → feature モード。複数あれば AskUserQuestion で対象の feature を選択
- plan.md がない場合 → standalone モード。そのまま Step 1 へ

### 0-b. feature モード: 過去の調査確認

```
Glob docs/plans/{feature-name}/troubleshoot-*.md
```

過去の調査ドキュメントがある場合:

1. 直近のドキュメントを Read し、前回の調査結果・修正方針を把握
2. AskUserQuestion でユーザーに確認:
   - 「前回の修正（{前回の修正方針の要約}）が効かなかったので再調査したい」
   - 「前回とは別の不具合を調査したい」
3. 前回の修正が効かなかった場合 → 前回の仮説が誤っていた可能性を含めて再調査
4. 別の不具合の場合 → 新規調査フローへ

出力ファイル名の連番: `troubleshoot-{YYYY-MM-DD}-{N}.md`
- 当日の既存ファイル数を確認し、次の番号を採番（初回は 1）

---

## Step 1: 症状の整理 + パターン判定

### 1-a. ヒアリング

$ARGUMENTS があればそこから、なければ AskUserQuestion で:
- 何をしたか（操作手順）
- 何が起きるはずだったか
- 実際に何が起きたか
- エラーメッセージ、ログがあれば

### 1-b. パターン判定

ヒアリング結果から不具合パターンを判定:

| パターン | 判定基準 | 調査方針 |
|---------|---------|---------|
| **A vs B 比較** | 「X だと動くが Y だと動かない」 | 両方の実行パスをトレースし分岐点を特定 |
| **期待 vs 実際** | 「X のはずが Y になる」 | 期待される実行パスを追い、値が変わる箇所を特定 |
| **エラー・クラッシュ** | エラーメッセージ、例外、無反応 | エラー発生箇所からスタックを遡る |
| **不安定** | 「たまに起きる」「時々動かない」 | タイミング依存の条件を洗い出す |
| **リグレッション** | 「前は動いてた」「X を変えたら壊れた」 | git diff と実行パスの交点を調べる |

パターンが明確に 1 つに当てはまらない場合は、最も近いものを選びつつ複合的に調査する。

AskUserQuestion でパターン判定結果を確認:
  質問文にパターン名と症状の整理を含める
  - 「この理解で合っている — 調査に進む」
  - 「修正がある」（→ ユーザーの修正を反映して再判定）

---

## Step 2: 実行フローのトレース

パターンに応じた調査を行う。**推測ではなく、コードを実際に読んで事実を確認する**ことが重要。

### 2-a. コードパスの調査（サブエージェント委譲）

```
Task(subagent_type: Explore) — code-researcher:
  「以下の機能の実行フローを、エントリポイントから最終出力まで
  コードを追って完全にトレースしてください。

  機能: {症状に関連する機能}

  報告形式:
  1. イベント/トリガーの起点
  2. 各ステップの処理内容（ファイル:行番号）
  3. 分岐条件（if/match）とその条件値
  4. 最終的な出力/副作用」
```

### 2-b. パターン別の追加調査

| パターン | 追加調査 |
|---------|---------|
| **A vs B 比較** | 動く場合 / 動かない場合の**両方**の実行パスをトレース。どこまで同じでどこから分かれるかを特定 |
| **期待 vs 実際** | 期待される値がどこで生成され、どこで変換/加工されるかをデータフローで追跡 |
| **エラー・クラッシュ** | エラーメッセージ / スタックトレースから発生箇所を特定し、そこに至る条件を逆向きに追跡 |
| **不安定** | 非同期処理・タイマー・共有状態・イベント順序に注目。レースコンディションの候補を列挙 |
| **リグレッション** | `git log --oneline` + `git diff` で関連する変更を特定し、変更箇所と実行パスの交点を調査 |

### 2-c. 環境・設定の確認

コードだけでは説明がつかない場合、以下も確認する:
- 環境変数・設定ファイルの値
- パッケージバージョン・依存関係の整合性
- DB の状態・データの整合性

```bash
# 例: 環境変数の確認
env | grep {関連キーワード}

# 例: パッケージバージョンの確認
cat package.json | grep {パッケージ名}
```

### 2-d. 実際の値の確認（推測の排除）

コード読解だけでは判断できない場合、実行時の実際の値を確認する:

- **テストを動かして確認**: 既存テストの実行、または検証用テストスクリプトの作成
- **一時デバッグログを追加** し、ユーザーに実行 → 結果報告を依頼
- 既存のログ出力があればその確認を依頼
- Bash でプロセスの状態確認、DB の中身確認など

デバッグログは調査完了後に除去する（Step 4 の修正方針に含める）。

---

## Step 3: 原因箇所の特定

Step 2 の調査結果から、**不具合の原因となっている正確な箇所**を特定する。

提示形式:

```
■ 原因の特定

実行フロー:
  1. {ステップ} （ファイル:行番号）  ← ✓ ここまで正常
  2. {ステップ} （ファイル:行番号）  ← ✓
  3. {ステップ} （ファイル:行番号）  ← ★ ここで問題発生
  4. {ステップ} （ファイル:行番号）  ← ✗ 以降が不正

原因: {なぜこの箇所で問題が起きるかの説明}
根拠: {コード調査 or ログ確認で得られた事実}
```

原因が特定できない場合は、**追加調査**（ログ追加 → 再実行 → 結果確認のサイクル）をユーザーと繰り返す。推測で先に進まない。

---

## Step 4: 修正方針の提示 + 調査ドキュメント出力

原因が特定された段階で、初めて修正方針を提示する。

### 4-a. ユーザーへの修正方針提示

```
■ 修正方針

原因: {Step 3 で特定した原因}

修正案:
  - {具体的な修正内容}（{ファイル:行番号}）

根拠: {なぜこの修正で直るか — 推測ではなく Step 2-3 の調査結果に基づく}

影響範囲: {この修正が他の動作に影響しないか}

次のアクション:
  - progress.md がある場合 → `/implement` で修正
  - standalone の場合 → 直接修正
```

修正案は**必ず原因からの論理的帰結**であること。「とりあえず試してみる」は禁止。

### 4-b. 調査ドキュメントの出力

#### feature モードの場合

調査結果を `docs/plans/{feature-name}/troubleshoot-{YYYY-MM-DD}-{N}.md` に Write で保存する。

```markdown
# 不具合調査: {症状の一行サマリー}

調査日: {YYYY-MM-DD}

## 関連する過去の調査

{過去の調査ドキュメントがある場合}
- 前回: [troubleshoot-{YYYY-MM-DD}-{N-1}.md](./troubleshoot-{YYYY-MM-DD}-{N-1}.md)
- 前回の修正方針: {前回の修正方針の要約}
- 前回の修正が効かなかった理由: {今回の調査で判明した理由}

{過去の調査ドキュメントがない場合}
なし（初回調査）

## 症状

パターン: {判定されたパターン}

{症状の詳細}

## 原因

{Step 3 の内容}

実行フロー:
  1. {ステップ} （ファイル:行番号）  ← ✓
  2. {ステップ} （ファイル:行番号）  ← ★ 問題箇所
  ...

根拠: {事実に基づく根拠}

## 修正方針

### コード修正
- {具体的な修正内容}（{ファイル:行番号}）

### plan.md の修正（該当する場合）
- {仕様の考慮漏れや修正が必要な箇所}

### デバッグログの除去
- {追加したデバッグログがあればここにリスト}

## 次のアクション
- `/implement` で修正（progress.md がある場合）
- 直接修正（standalone の場合）
```

#### standalone モードの場合

ドキュメント保存はしない（ユーザーが希望すれば任意の場所に保存）。
修正方針の提示（4-a）で調査は完了。

### 4-c. progress.md との連携（feature モードのみ）

progress.md が存在する場合、作業ログに以下を追記する:
- `troubleshoot 実施 → troubleshoot-{YYYY-MM-DD}-{N}.md 参照`
