# 仕様検証（Verify）

実装コードを **直接読み取り**、plan.md との乖離を双方向で検出する。
verifier エージェントが実コードと仕様書を突合し、検証の副産物として implementation-summary.md を生成する。

入力: `docs/plans/{feature-name}/plan.md` + 実装コード（git diff or ファイル一覧）
出力: 検証レポート + `implementation-summary.md`（検証結果から生成）

**パスルール**: `docs/plans/{feature-name}/` はカレントディレクトリ直下。`{feature-name}` は英語の kebab-case（例: `campaign-notification`）。パス区切り不可

## ワークフロー

```
Step 0: plan.md + 変更範囲の特定 + スコーピング
Step 1: 仕様と実装の直接突合（ドメイン別・重要度付き）
Step 2: ファイル構成の突合
Step 3: 受入条件の検証
Step 4: 実装サマリ生成 + 検証レポート + 次のアクション提示
```

---

## 重要度レベル

全ステップの検出結果に以下の重要度を付与する:

| レベル | 基準 | 例 |
|--------|------|-----|
| **Critical** | 仕様の主要機能が未実装、または根本的に異なる | 仕様に記述された操作が丸ごと未実装、必要なデータモデルが未作成、受入条件の未充足 |
| **Warning** | 仕様との部分的な不一致。動作はするが仕様通りではない | 仕様の入出力要件の部分的な不足、バリデーションルール漏れ、テストなし |
| **Info** | 軽微な差異。仕様の意図は満たしているが記述と異なる | 命名の微差、実装アプローチの違い（結果は同等）、仕様外の追加実装 |

---

## Step 0: plan.md + 変更範囲の特定

スキル起動直後に、対象 feature を特定する:

```
Glob docs/plans/**/plan.md
```

$ARGUMENTS に feature-name が指定されている場合はそのディレクトリを使用。
複数の仕様書ディレクトリがある場合はユーザーに選択を求める。

```
Read docs/plans/{feature-name}/plan.md
```

### 0-a. progress.md によるスコーピング

```
Read docs/plans/{feature-name}/progress.md
```

progress.md が存在する場合、タスク進捗テーブルから検証範囲を決定する:

- **全タスク `✓`** → **フル検証**: 全セクション + 受入条件を検証
- **一部タスク `✓`** → **部分検証**: 完了タスクの対象ファイル・関連セクションのみ検証。未完了分は「検証対象外」と明示
- **全タスク `-`** → 検証対象なし。「実装が開始されていません」と案内して終了

progress.md が存在しない場合はフル検証として扱う（strategy を経由していないケース）。

### 0-b. 実装差分の取得

```
Bash: git diff main --name-only
```

main ブランチとの差分がない場合は、ユーザーに比較対象のブランチ/コミットを確認する。

### 0-c. 検証範囲の提示

```
plan.md と実装差分を読み込みました:
- 機能: {feature summary}
- 検証モード: {フル検証 / 部分検証（{完了タスク数}/{全タスク数}）}
- 検証対象セクション: {バックエンド / DB / フロントエンド — 該当するもの}
- 変更ファイル: {N}ファイル

検証を開始します。
```

## Step 1: 仕様と実装の直接突合

verifier エージェントが **実装コードを直接 Read** し、plan.md の各ドメインセクションと突合する。
summary ファイルへの中間依存を排除し、実コードと仕様の直接比較で正確性を確保する。

### 適応的実行

変更規模に応じて実行方法を切り替える:

- **小規模**（変更ファイル 5 個以下、またはドメイン 1 つのみ）: 単一の verifier で全ドメインを一括検証
- **中〜大規模**（変更ファイル 6 個以上 & ドメイン 2 つ以上）: ドメイン別に並列実行（1-a, 1-b, 1-c）

### 1-a: バックエンド仕様突合（「バックエンド変更」セクションがある場合）

```
Task(subagent_type: implementation-verifier):
  プロンプト: 「仕様と実装の突合検証を行ってください。

  チェック観点: API（バックエンド）
  仕様書: docs/plans/{feature-name}/plan.md の「バックエンド変更」セクション
  実装ファイル: {対象ファイル一覧}

  注意: 仕様書は機能仕様書であり、具体的なエンドポイントパスや型定義は含まれていません。
  仕様に記述された「どんな操作を提供するか」「入出力は何か」「エラーケース」を、
  実装コードが機能的に満たしているかを検証してください。
  検証結果に加え、実装されたAPI操作の処理フロー概要（実装サマリ）も報告してください。」
```

### 1-b: DB仕様突合（「DB変更」セクションがある場合）

```
Task(subagent_type: implementation-verifier):
  プロンプト: 「仕様と実装の突合検証を行ってください。

  チェック観点: DB
  仕様書: docs/plans/{feature-name}/plan.md の「DB変更」セクション
  実装ファイル: {対象ファイル一覧（マイグレーション、スキーマ等）}

  注意: 仕様書は機能仕様書であり、具体的なテーブル名やカラム定義は含まれていません。
  仕様に記述された「どんなデータを保持するか」「データ間の関係性」「ビジネス上の制約」を、
  実装コードが機能的に満たしているかを検証してください。
  検証結果に加え、実装されたデータ構造と変更内容の概要（実装サマリ）も報告してください。」
```

### 1-c: フロントエンド仕様突合（「フロントエンド変更」セクションがある場合）

```
Task(subagent_type: implementation-verifier):
  プロンプト: 「仕様と実装の突合検証を行ってください。

  チェック観点: フロントエンド
  仕様書: docs/plans/{feature-name}/plan.md の「フロントエンド変更」セクション
  実装ファイル: {対象ファイル一覧}

  注意: 仕様書は機能仕様書であり、具体的なコンポーネント名やProps型は含まれていません。
  仕様に記述された「どんな画面/UIが必要か」「ユーザーが行える操作」「表示するデータ」を、
  実装コードが機能的に満たしているかを検証してください。
  ワイヤーフレームがある場合は、レイアウトの意図が実装に反映されているかも確認してください。
  検証結果に加え、実装された画面構成とUI要素の概要（実装サマリ）も報告してください。」
```

**並列実行**: 1-a, 1-b, 1-c は互いに独立しているため、中〜大規模の場合は並列で実行する。

## Step 2: ファイル構成の突合

plan.md の各ドメインセクションのファイル情報と、git diff の変更ファイル一覧を突合する。

**部分検証時**: 完了タスクの対象ファイルのみを突合対象とする。

**チェック項目**:
- 仕様書で「新規」とされたファイルが実際に作成されているか
- 仕様書で「変更」とされたファイルが実際に変更されているか
- 仕様書に記載のないファイルが変更されていないか（意図しない変更の検出）

**重要度の適用**:
- 仕様のファイルが未作成/未変更 → Critical
- 仕様外のファイルが変更 → Info（リファクタリングや依存更新の可能性）

**出力**:
```
## ファイル構成の突合

| 仕様書のファイル | 操作 | 実際の状態 | 重要度 | 判定 |
|---------------|------|---------|--------|------|
| {ファイルパス} | 新規 | ✓ 作成済み | - | OK |
| {ファイルパス} | 変更 | ✗ 未変更 | Critical | NG |
| {ファイルパス} | - | 変更あり | Info | 要確認 |
```

---

## Step 3: 受入条件の検証

plan.md の受入条件を1つずつ検証する。

**フル検証時**: 全受入条件を検証。
**部分検証時**: 完了タスクに関連する受入条件のみ検証。plan.md のテスト方針にトレーサビリティ表があればそれを参照して対応を判定する。

**検証方法**:
- 各受入条件について、対応するテストコードまたは実装コードを Grep/Read で確認
- テストコードがある場合: テストの内容が受入条件を満たしているか確認
- テストコードがない場合: 実装コードから受入条件の充足を判断

**重要度の適用**:
- 実装もテストもなし → Critical
- 実装はあるがテストなし → Warning
- 実装もテストもあり → 重要度なし（PASS）

**出力**:
```
## 受入条件の検証

| # | 受入条件 | テスト | 実装 | 重要度 | 判定 |
|---|---------|-------|------|--------|------|
| 1 | {受入条件} | ✓ {テストファイル} | ✓ 実装済み | - | PASS |
| 2 | {受入条件} | ✗ テストなし | ✓ 実装済み | Warning | PARTIAL |
| 3 | {受入条件} | - | ✗ 未実装 | Critical | FAIL |
```

---

## Step 4: 実装サマリ生成 + 検証レポート + 次のアクション提示

### 4-a. implementation-summary.md の生成

Step 1 の verifier 出力に含まれる実装サマリを統合し、`docs/plans/{feature-name}/implementation-summary.md` を Write する。

**テンプレート**（変更種別に応じて該当セクションのみ記述）:

```markdown
# {機能名} — 実装サマリ（Implementation Summary）

> 生成日: {日付}
> 検証モード: {フル検証 / 部分検証}

## 機能概要

{plan.md の概要を元に、実装の実態を1-3文で記述}

## 処理フロー

{verifier の出力から、主要な処理フローを記述}
{API操作がある場合: エンドポイント単位で リクエスト→バリデーション→ロジック→レスポンス}
{フロントエンドのみ: コンポーネント単位で 入力→状態管理→表示}
{バッチ/ジョブ: ジョブ単位で トリガー→処理→完了}

## 技術サマリー

- **API**: {追加/変更されたエンドポイント}（該当する場合）
- **DB**: {追加/変更されたテーブル}（該当する場合）
- **UI**: {追加/変更された画面/コンポーネント}（該当する場合）
```

### 4-b. 検証レポートの生成

Step 1〜3 の結果を統合し、最終判定を行う。

**判定基準**:
- **PASS**: Critical 0件 & Warning 0件
- **PARTIAL**: Critical 0件、Warning のみ
- **NEEDS_FIX**: Critical 1件以上

**レポート出力**:
```
## 検証レポート

### 総合判定: {PASS / PARTIAL / NEEDS_FIX}

### サマリー
- 検証モード: {フル検証 / 部分検証（{完了タスク数}/{全タスク数}）}
- ファイル構成: {OK数}/{総数}
- バックエンド仕様: {PASS/PARTIAL/NEEDS_FIX}（Critical {N} / Warning {N} / Info {N}）
- DB仕様: {PASS/PARTIAL/NEEDS_FIX}（Critical {N} / Warning {N} / Info {N}）
- フロントエンド仕様: {PASS/PARTIAL/NEEDS_FIX}（Critical {N} / Warning {N} / Info {N}）
- 受入条件: {PASS数}/{総数}

### 検出事項（重要度順）

#### Critical
| # | 分類 | 方向 | 内容 | 対応方法 |
|---|------|------|------|---------|
| 1 | {分類} | 仕様→実装 | {内容} | {対応方法} |

#### Warning
| # | 分類 | 方向 | 内容 | 対応方法 |
|---|------|------|------|---------|
| 2 | {分類} | 実装→仕様 | {内容} | {対応方法} |

#### Info
| # | 内容 |
|---|------|
| 3 | {内容} |
```

該当する検出事項がないレベルのセクションは省略する。

### 4-c. 次のアクション提示

- **PASS の場合**: 「検証完了。PR作成/更新に進めます。」
- **PARTIAL の場合**:
  - 実装修正が必要 → 「以下の Warning 項目の修正を推奨します: {修正項目}」
  - 仕様修正が必要 → 「plan.md を直接編集して仕様を修正してください」
- **NEEDS_FIX の場合**:
  - 実装漏れ（仕様→実装） → 「以下の実装が不足しています: {不足項目}」
  - 仕様不足（実装→仕様） → 「plan.md を直接編集して仕様を更新してください」
  - 型不一致/ロジック差異 → 「仕様と実装に差異があります。どちらに合わせるか確認してください。」
- **部分検証の場合**: 上記に加えて「残り {N} タスクの実装完了後、フル検証を実行してください。」
