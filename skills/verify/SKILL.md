# 仕様検証（Verify）

実装コードからデータフローを抽出して実装サマリを生成し、plan.md との乖離を検出する。
**code-researcher** エージェントでデータフロー抽出 → **implementation-verifier** エージェントで plan.md と双方向突合を行う。

入力: `docs/{feature-name}/plan.md` + 実装コード（git diff or ファイル一覧）
出力: `implementation-summary.md` + 検証レポート + 次のアクション提示

**パスルール**: `docs/{feature-name}/` はカレントディレクトリ直下。`{feature-name}` は日本語のシンプルな名前（パス区切り不可）

## ワークフロー

```
Step 0: plan.md + git diff の読み込み
Step 1: 実装サマリの生成 ← code-researcher でデータフロー抽出 → implementation-summary.md を Write
Step 2: 実装サマリ vs plan.md の乖離検出 ← implementation-verifier に委譲（セクション別・双方向）
Step 3: ファイル構成の突合
Step 4: 受入条件の検証
Step 5: 検証レポート生成 + 次のアクション提示
```

---

## Step 0: plan.md + git diff の読み込み

スキル起動直後に、対象 feature を特定する:

```
Glob docs/**/plan.md
```

$ARGUMENTS に feature-name が指定されている場合はそのディレクトリを使用。
複数の仕様書ディレクトリがある場合はユーザーに選択を求める。

```
Read docs/{feature-name}/plan.md
```

**progress.md の読み込み**（任意）:
`docs/{feature-name}/progress.md` が存在する場合は Read し、完了タスクの情報を検証の参考とする。

**実装差分の取得**:
```
Bash: git diff main --name-only
```

mainブランチとの差分がない場合は、ユーザーに比較対象のブランチ/コミットを確認する。

読み込んだ内容をユーザーに提示:

```
plan.md と実装差分を読み込みました:
- 機能: {feature summary}
- 仕様セクション: {存在するセクション一覧}
- 変更ファイル: {N}ファイル

検証を開始します。
```

---

## Step 1: 実装サマリの生成

**code-researcher** エージェントに委譲し、実装コードからデータフローを抽出する。
抽出結果を `docs/{feature-name}/implementation-summary.md` として Write する。

### 1-1: データフローの抽出

```
Task(code-researcher) を起動:
  プロンプト: 「以下の実装コードを読み、API単位の処理フロー（データフロー）を要約してください。
  対象ファイル: {git diff で検出された実装ファイル一覧}
  出力形式: API操作ごとに以下の流れを箇条書き:
    1. クライアントからのリクエスト（メソッド、パス、主要パラメータ）
    2. バリデーション（主要なチェック内容）
    3. DB操作やビジネスロジック
    4. レスポンス（ステータスコード、レスポンス概要）
  追加抽出項目:
    - 認証・認可チェック（ミドルウェア、ガード等）
    - DB変更（テーブル追加・変更、マイグレーション内容）
    - フロントエンド変更（コンポーネント、Props型、Hooks、状態管理）
  ビジネスロジックの骨格がわかる粒度で記述してください。」
```

**フロントエンドのみの変更時**: API操作ではなくコンポーネント単位で抽出する。

### 1-2: 実装サマリの生成

code-researcher の出力を整形し、`docs/{feature-name}/implementation-summary.md` を Write する。

**テンプレート**:

```markdown
# {機能名} — 実装サマリ（Implementation Summary）

> 生成日: {日付}

## 機能概要

{plan.md の概要セクションを元に、実装後の実態を1-3文で記述}

## データフロー

### {操作名（例: リマインダー作成）}
1. クライアント → {メソッド} {パス}（{主要パラメータ}）
2. 認証: {認証・認可チェックの内容}
3. バリデーション: {主要なチェック内容}
4. {DB操作やビジネスロジック}
5. → {ステータスコード}（{レスポンス概要}）

## 技術サマリー

- **API**: {追加/変更されたエンドポイントの箇条書き}
- **DB**: {追加/変更されたテーブルの箇条書き}
- **UI**: {追加/変更された画面/コンポーネントの箇条書き}
```

---

## Step 2: 実装サマリ vs plan.md の乖離検出

`implementation-summary.md` と plan.md を入力として、**implementation-verifier** エージェントに委譲してセクション別・双方向の乖離検出を行う。

plan.md に存在するドメインセクションに応じて、以下を並列で実行する。

### 2-a: バックエンド仕様突合（「バックエンド変更」セクションがある場合）

```
Task(implementation-verifier) を起動:
  プロンプト: 「バックエンド仕様の突合検証を行ってください。
  仕様書: docs/{feature-name}/plan.md の「バックエンド変更」セクションを参照
  実装サマリ: docs/{feature-name}/implementation-summary.md を Read して参照
  チェック観点: API
  実装ファイル: {plan.md のバックエンド変更セクションに記載されたファイル一覧}
  検証方向:
  (A) 実装→仕様: 実装サマリの各API操作が仕様に定義されているか
  (B) 仕様→実装: 仕様の各エンドポイント定義が実装サマリに反映されているか
  検証項目:
  - エンドポイントのメソッド・パスの一致
  - リクエスト/レスポンス型のフィールドの一致
  - バリデーションルールの実装
  - エラーコード・ステータスコードの一致
  - 認証・認可チェックの実装」
```

### 2-b: DB仕様突合（「DB変更」セクションがある場合）

```
Task(implementation-verifier) を起動:
  プロンプト: 「DB仕様の突合検証を行ってください。
  仕様書: docs/{feature-name}/plan.md の「DB変更」セクションを参照
  実装サマリ: docs/{feature-name}/implementation-summary.md を Read して参照
  チェック観点: DB
  実装ファイル: {plan.md のDB変更セクションに記載されたファイル一覧}
  検証方向:
  (A) 実装→仕様: 実装サマリのDB変更が仕様に定義されているか
  (B) 仕様→実装: 仕様のテーブル定義が実装サマリに反映されているか
  検証項目:
  - テーブル名・カラム名の一致
  - カラムの型・制約の一致
  - インデックスの定義
  - マイグレーションの内容
  - 外部キー・リレーションの一致」
```

### 2-c: フロントエンド仕様突合（「フロントエンド変更」セクションがある場合）

```
Task(implementation-verifier) を起動:
  プロンプト: 「フロントエンド仕様の突合検証を行ってください。
  仕様書: docs/{feature-name}/plan.md の「フロントエンド変更」セクションを参照
  実装サマリ: docs/{feature-name}/implementation-summary.md を Read して参照
  チェック観点: フロントエンド
  実装ファイル: {plan.md のフロントエンド変更セクションに記載されたファイル一覧}
  検証方向:
  (A) 実装→仕様: 実装サマリの各コンポーネント/Hooksが仕様に定義されているか
  (B) 仕様→実装: 仕様の各コンポーネント定義が実装サマリに反映されているか
  検証項目:
  - コンポーネントの作成
  - Props型のフィールドの一致
  - Hooks のシグネチャの一致
  - エッジケースのハンドリング」
```

**並列実行**: 2-a, 2-b, 2-c は互いに独立しているため、可能な限り並列で実行する。

---

## Step 3: ファイル構成の突合

plan.md の各ドメインセクションのファイル情報と、git diff の変更ファイル一覧を突合する。

**チェック項目**:
- 仕様書で「新規」とされたファイルが実際に作成されているか
- 仕様書で「変更」とされたファイルが実際に変更されているか
- 仕様書に記載のないファイルが変更されていないか（意図しない変更の検出）

**出力**:
```
## ファイル構成の突合

| 仕様書のファイル | 操作 | 実際の状態 | 判定 |
|---------------|------|---------|------|
| {ファイルパス} | 新規 | ✓ 作成済み | OK |
| {ファイルパス} | 変更 | ✗ 未変更 | NG |
| {ファイルパス} | - | 変更あり | 要確認 |
```

---

## Step 4: 受入条件の検証

plan.md の受入条件を1つずつ検証する。

**検証方法**:
- 各受入条件について、対応するテストコードまたは実装コードを Grep/Read で確認
- テストコードがある場合: テストの内容が受入条件を満たしているか確認
- テストコードがない場合: 実装コードから受入条件の充足を判断

**出力**:
```
## 受入条件の検証

| # | 受入条件 | テスト | 実装 | 判定 |
|---|---------|-------|------|------|
| 1 | {受入条件} | ✓ {テストファイル} | ✓ 実装済み | PASS |
| 2 | {受入条件} | ✗ テストなし | ✓ 実装済み | PARTIAL |
| 3 | {受入条件} | - | ✗ 未実装 | FAIL |
```

---

## Step 5: 検証レポート生成 + 次のアクション提示

Step 1〜4 の結果を統合し、最終判定を行う。

**判定基準**:
- **PASS**: 全受入条件充足、ファイル構成一致、重大な仕様不一致なし
- **PARTIAL**: 軽微な不一致はあるが受入条件は概ね充足
- **NEEDS_FIX**: 受入条件の未充足、またはドメイン仕様との重大な不一致あり

**レポート出力**:
```
## 検証レポート

### 総合判定: {PASS / PARTIAL / NEEDS_FIX}

### サマリー
- ファイル構成: {OK数}/{総数}
- データフロー乖離: {N}件（実装→仕様: {N}件、仕様→実装: {N}件）
- バックエンド仕様: {PASS/NEEDS_FIX}（{不一致数}件）
- DB仕様: {PASS/NEEDS_FIX}（{不一致数}件）
- フロントエンド仕様: {PASS/NEEDS_FIX}（{不一致数}件）
- 受入条件: {PASS数}/{総数}

### 要対応項目（NEEDS_FIX / PARTIAL の場合）
| # | 分類 | 方向 | 内容 | 対応方法 |
|---|------|------|------|---------|
| 1 | {分類} | 実装→仕様 | {内容} | {対応方法} |
```

**次のアクション提示**:

- **PASS の場合**: 「検証完了。PR作成/更新に進めます。」
- **PARTIAL の場合**:
  - 実装修正が必要 → 「以下の項目を修正してください: {修正項目}」
  - 仕様修正が必要 → 「`/feature-spec:revise` で仕様を修正してください」
- **NEEDS_FIX の場合**:
  - 実装漏れ（仕様→実装） → 「以下の実装が不足しています: {不足項目}」
  - 仕様不足（実装→仕様） → 「`/feature-spec:revise` で仕様を更新してください」
  - 型不一致/ロジック差異 → 「仕様と実装に差異があります。どちらに合わせるか確認してください。」
