# Skill SKILL.md 編集ルール

## Frontmatter 制約

- **name**: 小文字+ハイフンのみ。ディレクトリ名と一致させること。最大64文字、XMLタグ禁止、予約語（`anthropic`, `claude`）禁止
- **命名パターン**: ジェランド形を推奨（例: `processing-pdfs`, `generating-reports`）
- **description**: 1024文字以内。三人称で書き始め、「何をするか」→「いつ使うか」の順。"I can" / "You can" は禁止
- **allowed-tools**: スペース区切りで列挙（例: `Read Glob Grep Edit Task AskUserQuestion`）
- **metadata.triggers**: スキル発動のトリガーワードをカンマ区切りで記述

## Description の書き方

- 三人称で開始する（例: "Generates ...", "Analyzes ..."）
- 1文目: 何をするか（機能）
- 2文目: いつ使うか（ユースケース）
- 主語に "I" や "You" を使わない

## 本文の制約

- **500行以下** に収める。超える場合は `references/` ディレクトリに分割する
- Claude が既に知っている一般知識は書かない。トークンコストを意識する
- 同じ概念には一貫した用語を使う。用語のブレは混乱を招く
- 時限的な情報（バージョン番号、日付依存の記述）は避ける

## 指示の自由度

- **高い自由度**: 創造的タスク（文章生成、デザイン提案）→ ゴールだけ示し手段は委ねる
- **中程度**: 一般的タスク（コード生成、リファクタ）→ ステップと制約を提示
- **低い自由度**: 安全性が重要なタスク（データ操作、本番デプロイ）→ 手順を厳密に指定

## ワークフロー設計

- 複雑タスクはステップに分割し、チェックリスト形式で記述する
- 各ステップの完了条件を明確にする
- **フィードバックループ**: バリデーション → 修正 → 再バリデーション のサイクルを組み込む
- ループの最大回数を設定して無限ループを防ぐ

## パターン

- **テンプレート**: 入力→出力のペアを例示して期待形式を伝える
- **例示**: 良い例・悪い例を併記し、品質基準を具体化する
- **デフォルト提示**: 推奨1つ + 代替はエスケープハッチとして提供。選択肢を広げすぎない

## 参照ファイル

- SKILL.md から直接参照する（1階層）。ネストした参照チェーンは避ける
- 参照先はスキル自身の `references/` 配下か、エージェント側の `references/` を指す
- 100行超のファイルは先頭に目次（セクション一覧）を付ける
- パスはフォワードスラッシュのみ使用（`\` は禁止）
