# リサーチ: リアーキテクチャ方向性（SQLite 以外）

調査日: 2026-03-14
調査タイプ: 探索的調査（analyzer → researcher）

## 調査ゴール
現行の md ファイルベースアーキテクチャの課題を解決するリアーキテクチャの選択肢（SQLite 移行以外）とそのトレードオフを明らかにする

## 現状

### アーキテクチャ概要
- 6スキル（spec, build, check, fix, research, list）+ 4エージェント（analyzer, writer, verifier, researcher）
- データ: `docs/plans/{feature}/` 配下に plan.md, progress.md, result.md, research-*.md, debug-*.md
- Annotation Viewer: Python 3 標準ライブラリ HTTP サーバー + シングルファイル SPA
- フェーズ判定: ファイル存在ベースの推論

### 主要な課題
| 課題 | 詳細 |
|------|------|
| 共通定義の不在 | パスパターン・ステータス判定が各スキルにインライン実装（6スキルが同じパスをハードコード） |
| フォーマット依存パース | progress.md の Markdown テーブルを正規表現でパース（build, check, list で重複） |
| ポーリング通信 | spec の Annotation Cycle が `sleep 2` ループでファイル監視 |
| 線形コスト増加 | 22プラン蓄積済み。list は全 plan.md + progress.md + result.md を並列 Read |
| テスト不在 | スキル/エージェントの単体・統合テストがない |

## 調査結果

|------|--------------------|-----------------------------|--------------|---------|--------------|----------------|

### 発見事項

#### A. 現行改善（md ファイルのまま構造化強化）

- 事実: LLM のフォーマット理解研究では Markdown は JSON より 34〜38% 少ないトークンで同等精度（出典: improvingagents.com）
- 事実: 公式 best practices で「ファイルは1階層の参照にとどめる」「domain-specific organization で共通参照を整理する」と推奨（出典: platform.claude.com）
- 示唆: パス重複は `agents/shared/paths.md` に共通定義を置くだけで解決可能。アーキテクチャ変更は不要かもしれない
- トレードオフ: プラン数の線形コスト・ポーリングは本質的に解決しない

#### B. YAML フロントマターへの状態集約

- 事実: YAML frontmatter は Hugo/Jekyll/Gatsby で実績あるパターン（出典: blog.tech4teaching.net）
- 示唆: progress.md の frontmatter に `current_status: in_progress` の1行を追加するだけで、list は Grep のみでステータス取得可能になる。22プランへの変更は最小限
- トレードオフ: YAML はインデントミスで壊れやすい。配列のネストで可読性低下

#### C. JSONL によるイベントログ

- 事実: JSONL は追記のみで部分書き込みが corrupted state を残さない（出典: Oracle Developers）
- 示唆: md は人間向け、JSONL は機械向けの二重管理が中間解として機能する。Claude は最終行だけ読めば現在状態を確認できる
- トレードオフ: 移行スクリプトが必要。Claude が Edit で末尾追加するパターンの指示が増える

#### D. MCP サーバー型

- 事実: MCP はローカル STDIO で動作可能。ツールは1つの明確な目的を持つべき（出典: modelcontextprotocol.io）
- 示唆: パスパターン・ステータス判定ロジックを MCP サーバー側に閉じ込められる
- トレードオフ: `mcp` SDK が必要で「ゼロ外部依存」を破る。ライフサイクル管理が新たな複雑性に

#### E. SSE への移行

- 事実: SSE は `Content-Type: text/event-stream` で Python 標準ライブラリのみで実装可能（出典: fictionally-irrelevant.vercel.app）
- 事実: server.py は既に `EVENT:comments_saved:{feature}` を stdout 出力しており、SSE への変換コストは低い
- トレードオフ: Claude Bash ツールのタイムアウト制限で長時間 SSE 待機が中断される可能性

#### F. Git ネイティブ（git notes）

- 事実: git notes はコミット SHA を変更せずメタデータを追記できる。Gemini Conductor 等の活用事例あり（出典: dariuszparys.com）
- トレードオフ: list スキルの `allowed-tools` に Bash がないため使用不可。ブランチ切替時の文脈喪失問題

#### G. スクリプト委譲パターン

- 事実: 公式 best practices で「Claude が生成するコードより事前に書かれたスクリプトの方が信頼性が高い」と明示（出典: platform.claude.com）
- 示唆: ステータス判定ロジックを `scripts/get-plan-status.py` に切り出し、スキルは `Bash: python3 scripts/get-plan-status.py {feature}` を呼ぶだけにする設計は、Markdown パース依存を解消しつつ外部依存を増やさない

## 探索の視点

### 「本当にファイルベースが問題か？」

- 事実: LLM のファイルシステム研究では「モデルはすでにディレクトリ一覧、grep、範囲読み取りを知っており、ファイルシステムはインターフェースとして勝利している」（出典: Oracle Developers）
- 示唆: 問題の本質はファイルベースそのものではなく、**パース対象が「構造化されていない Markdown テーブル」であること**と**スキル間でロジックが重複していること**に絞られる可能性がある

### 課題と解決手段の対応

| 課題 | SQLite で解決 | SQLite なしで解決可能か |
|------|-------------|----------------------|
| パス重複 | ○（DB が一元管理） | ○ 共通参照 md を1箇所に置くだけ |
| Markdown テーブルパース | ○（構造化 DB） | ○ フロントマター集約 or スクリプト委譲 |
| ポーリング | ×（別問題） | △ SSE 移行（Bash タイムアウト制約あり） |
| 線形コスト | ○（SQL クエリ） | △ フロントマター Grep で大幅改善可能 |
| テスト不在 | ×（別問題） | ×（別問題） |

### 予想外のアプローチ

1. **server.py 側へのポーリング引き受け**: Claude 側の while ループを廃止し、server.py がファイル監視 → SSE 送信する構成。Claude 側のポーリングを排除できる可能性
2. **段階的リアーキテクチャ**: SQLite フル移行のような一括変更ではなく、(1) 共通参照 md 整備 → (2) フロントマター状態集約 → (3) スクリプト委譲 の段階的改善で、各ステップが独立してデプロイ可能

## 推奨・結論

**フル移行（SQLite 等）の前に、段階的改善で課題の大半を解決できる可能性が高い。**

推奨ステップ:
1. **共通参照 md の整備**（低コスト・高効果）: パスパターン・ステータス判定を1箇所に集約
2. **フロントマターへの状態集約**（低コスト・中効果）: list の線形コストを Grep ベースに改善
3. **スクリプト委譲**（中コスト・高効果）: Markdown パース依存を解消。公式推奨パターン
4. **SSE 移行の検証**（中コスト・効果要検証）: Claude Bash タイムアウト制約の実機検証が先

SQLite 移行は「上記の段階的改善では不十分」と判明した場合の最終手段として位置づけるのが妥当。
