# 技術調査: analyzer への確認事項・追加検討事項の収集指示最適化

調査日: 2026-03-13
調査タイプ: 技術調査

## 調査ゴール

spec SKILL.md の Step 2 で analyzer に渡すプロンプトに、確認事項（推測・仮定）と追加検討事項（仕様で触れていない影響観点）を高品質に検出させるための具体的な指示文を設計するための情報収集。

## 調査結果 1: tsumiki kairo-requirements.md の信号機判定方式

出典: https://raw.githubusercontent.com/classmethod/tsumiki/main/commands/kairo-requirements.md

### 信号機の定義

tsumiki は要件の「情報の出自」を3段階で表す:

| 信号 | 意味 | 条件 |
|------|------|------|
| 🔵 | ほぼ推測なし | PRD・EARS・設計文書・ユーザヒアリングを直接参照した |
| 🟡 | 妥当な推測 | 上記資料から論理的に導出可能 |
| 🔴 | 根拠なし推測 | 資料に根拠がなく推測した |

重要点: 信号機は「正確性の評価」ではなく「情報の出自（どこから来たか）」のメタ情報である。

### 確認事項の検出カテゴリ（tsumiki から抽出）

tsumiki は以下の5カテゴリでギャップを検出する:

1. **既存設計の妥当性・過不足**: 現行の制約（性能・データ制限）が現状に合っているか
2. **未定義・曖昧部分**: 入出力データの詳細仕様、UI/UX の期待値
3. **追加・変更要件**: 外部連携、レポート機能、運用要件など仕様に書かれていないもの
4. **影響範囲**: 新機能が既存のパフォーマンス・セキュリティ要件に与える影響
5. **優先順位・スコープ**: Must Have / Should Have の境界が不明確なもの

### 出典の記載形式

各項目に「どの資料から来た情報か」を付記させる:

```
*{出典タイプ}: {具体的な参照先}*
例: *PRD第2章より*, *既存DBスキーマより*, *設計文書から妥当な推測*
```

出典が存在しない場合 → 🔴を付与して「根拠となる資料がない推測」と明記。

## 調査結果 2: tsumiki kairo-design.md の信号機判定方式

出典: https://raw.githubusercontent.com/classmethod/tsumiki/main/commands/kairo-design.md

### requirements.md との共通点・相違点

- 定義は同一（🔵/🟡/🔴 の意味は同じ）
- 適用範囲が異なる: design フェーズでは型定義・DBスキーマ・APIエンドポイントの「各フィールド・各テーブル・各エンドポイント」単位に信号機を付ける
- 「すべての項目に信頼性レベルと出典を記載」が必須となっており、省略不可の設計

### spec-flow analyzer との対応

tsumiki の design フェーズが判定する確認事項の検出対象は、spec-flow の analyzer が担う「コードベース分析の結果として影響しそうな観点」に相当する:

- 技術選択の不確実性（フレームワーク・DB種別が暫定的）
- 制約の不明確さ（パフォーマンス・セキュリティ要件が未定義）
- 曖昧性（複数の実現方法が可能な設計項目）

## 調査結果 3: 確認事項を高品質に検出するための観点カテゴリ

tsumiki の分析 + 一般的なプロンプトエンジニアリングの知見から抽出。

### カテゴリ A: エッジケース

| 観点 | 検出トリガー |
|------|------------|
| 境界値 | 仕様に「N件まで」「N文字以内」がある場合、N+1 の扱いが未定義なら検出 |
| 空・null データ | 一覧表示の仕様があるが「0件時の表示」が未定義なら検出 |
| 大量データ | 検索・一覧機能があるがページネーション・上限が未定義なら検出 |
| 同時操作 | 複数ユーザーが同じリソースを操作する可能性がある機能で未定義なら検出 |

### カテゴリ B: 非機能要件

| 観点 | 検出トリガー |
|------|------------|
| パフォーマンス | N+1問題が発生しうるデータ取得パターンが実装に存在するが要件にない |
| セキュリティ | 認証済みルートへのアクセス制御が仕様に書かれていない |
| 認可 | ロール・権限による操作制限が仕様に書かれていない |
| データ保護 | 個人情報・センシティブデータの扱いが未定義 |

### カテゴリ C: 暗黙の前提（既存仕様との衝突）

| 観点 | 検出トリガー |
|------|------------|
| 既存 API 変更影響 | 既存エンドポイントのシグネチャを変更するが、他の呼び出し元が存在する |
| DB スキーマ影響 | カラム追加・変更が既存データのマイグレーション要否を伴う |
| 型定義の破壊的変更 | 既存の型定義を変更すると他のコンポーネントに影響が伝播する |
| ビジネスロジックの前提 | 既存の計算ロジックが「ある値は必ず正」等の前提を持っている |

### カテゴリ D: 技術的制約

| 観点 | 検出トリガー |
|------|------------|
| フレームワーク制約 | 使用フレームワークのバージョンで特定の実装方法が制限される |
| DB 制約 | トランザクション境界、インデックス、外部キー制約が要件と衝突 |
| 外部 API 制約 | レート制限・タイムアウト・ペイロードサイズ制限が仕様にない |
| ファイルシステム制約 | ファイルアップロード機能があるが容量・形式制限が未定義 |

## 調査結果 4: 現在の analyzer output.md フォーマットとの整合性分析

対象ファイル: `agents/analyzer/references/formats/output.md` L131-L147

### 現在の確認事項セクション定義

```markdown
## 確認事項

分析中に発見した推測・仮定を列挙する。仕様で明示されていないが実装時に確認が必要な事項。

| # | 項目 | 根拠 | ステータス |
|---|------|------|-----------|
| 1 | {確認が必要な事項} | {なぜ確認が必要か、コードや仕様のどこから推測したか} | {✅確認済み / ⚠️要確認 / ❓要議論} |
```

### 現在の追加検討事項セクション定義

```markdown
## 追加検討事項

仕様で触れられていないが、コードベース分析の結果として影響しそうな観点を列挙する。

- {観点}: {詳細説明}
```

### 整合性の評価

**強み:**
- 「根拠」列の設計は tsumiki の出典要求と同じ方向性
- ステータス（✅/⚠️/❓）は「情報の出自」ではなく「次のアクション」を示すメタ情報であり、analyzer の「事実報告」原則と整合する（既存 research ファイル L27 の懸念より）

**ギャップ:**
- 「何を検出すべきか」の観点カテゴリが analyzer への指示に存在しない。カテゴリがなければ検出品質がモデルの推論任せになる
- 「追加検討事項」と「確認事項」の使い分けが曖昧。両者の違いを明確にする指示が必要

### 両者の定義の違い（明確化）

| セクション | 対象 | 起点 |
|-----------|------|------|
| 確認事項 | 仕様の中の推測・仮定（仕様内の不確かな部分） | 機能概要・要件ヒアリング結果 |
| 追加検討事項 | 仕様が触れていない影響観点（仕様の外の話） | コードベース・既存実装の分析結果 |

## 設計への示唆

以下は調査結果から導出した観点であり、実装提案ではない。

### 示唆 1: analyzer への観点カテゴリの明示

現在の Step 2 プロンプト（SKILL.md L132-L133）は「確認事項を収集してください」という曖昧な指示のみ。tsumiki が5カテゴリ + 検出メカニズム（資料 → ギャップ分析 → 質問変換）を明示しているのと対照的。

検出品質を上げるには、analyzer に「何を見て」「何を確認事項と判断するか」の観点を渡す必要がある。

### 示唆 2: 根拠（出典）の記載を必須にする

tsumiki は「出典がない = 🔴」を強制することで、根拠なし推測の混入を防いでいる。spec-flow の「根拠」列も同様に、コードの具体的なファイルパス・行番号・仕様のどの記述から推測したかを必須にすることで品質が上がる。

### 示唆 3: 確認事項と追加検討事項の分離指示

現在は「また、分析の中で以下も収集してください」という並列指示。両者の起点（仕様内 vs コードベース）と用途（実装前の確認 vs 設計への考慮追加）の違いを指示文に明記すると、モデルが正しく分類できる。

### 示唆 4: 検出の「目安量」

tsumiki は「質問は一度に全部投げない」「段階的実施」を原則にしている。analyzer のケースでは逆に、検出が少なすぎると実装時のリスクが残る。フォーマットの注記に「検出ゼロの場合は省略可だが、エッジケース・非機能・暗黙前提・技術制約の4カテゴリを必ず検討してから判断すること」を加えることで、安易なスキップを防げる。
