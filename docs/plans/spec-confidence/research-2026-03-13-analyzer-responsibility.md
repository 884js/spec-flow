# リサーチ: analyzer の責務と確認事項の配置設計

調査日: 2026-03-13
調査タイプ: コードベース

## 調査ゴール
「確認事項（Assumptions）」「追加検討事項」の収集・提示・出力がどのレイヤーに属すべきかを判断する

## 現状

### 各コンポーネントの責務

| レイヤー | 責務の性質 | フォーマット制御 |
|---------|-----------|---------------|
| analyzer | 事実の収集・報告（「〜である」で記述。設計提案はしない） | `output.md` に追従 |
| writer | 構造化された入力をフォーマットに従い出力。判断しない | `formats/plan.md` + `examples/plan.md` に追従 |
| spec SKILL.md | オーケストレーション。ユーザー対話・データ受け渡し・フロー制御 | 対象外（直接記述） |

### フォーマット追従設計

- analyzer.md / writer.md 本体は「フォーマットファイルを Read してそれに従え」という汎用指示のみ
- セクション追加はフォーマットファイル側だけで完結する
- **制約**: 振る舞いの変更（判断ロジック）はフォーマットファイルでは制御できない

### 現在の実装で検出された問題

1. **analyzer に「✅/⚠️/❓ のステータス判定」をさせている**: analyzer の責務は「事実の報告」だが、ステータス判定は**評価**に踏み込んでいる。「推測・仮定を列挙する」は事実報告の延長として妥当だが、「確認済み/要確認/要議論」の判定は解釈を含む

2. **方向性確認ステップの削除**: ユーザー編集で旧 Step 3（方向性確認）が削除され、追加検討事項をユーザーに提示する場所が失われた（AC-4 未対応）

3. **データフローの断絶**: analyzer が「追加検討事項」を出力しても、spec SKILL.md にそれをユーザーに提示するロジックがない

## 発見事項

### analyzer の責務として妥当な範囲
- 「仕様で明示されていない推測・仮定」の列挙 → 事実報告の延長として妥当
- 「仕様で触れられていないが影響しそうな観点」の列挙 → コードベース分析の結果として妥当
- ステータス判定（✅/⚠️/❓）→ **評価に踏み込んでおり、analyzer の「事実報告のみ」原則と矛盾**

### spec SKILL.md が担うべき範囲
- 確認事項のステータス判定（ユーザーとの対話で確定）
- 追加検討事項のユーザー提示と確認
- 確認事項の writer への引き渡し
