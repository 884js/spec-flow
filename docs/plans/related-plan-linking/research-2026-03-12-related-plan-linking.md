# リサーチ: プラン間の関連リンク機構

調査日: 2026-03-12
調査タイプ: 複合

## 調査ゴール

plan.md に関連プランへのリンクを自動検出・配置する仕組みを設計する。配置場所と自動検出の方法を明らかにする。

## 現状（コードベース調査）

### 既存プランの関連性

10個のプランが存在し、以下の関連パターンが確認された:

| パターン | 例 |
|---------|-----|
| 進化関係（同一コンポーネントの段階的改善） | annotation-cycle → rich-markdown-preview → live-reload-preview |
| 依存関係（前提となるプラン） | spec-loop-redesign → readme-update, readme-and-fix-rename |
| 同一機能領域（同じスキルの異なるステップを拡張） | annotation-cycle と multi-research（共に spec スキル拡張） |
| 機能連携（実装面とドキュメント面の対） | multi-research と readme-research-step |

### 現在のフォーマットの状況

- plan.md フォーマット定義に「関連プラン」セクションは存在しない
- `参考資料` セクション（末尾）が最も近いが、外部URL向けの設計
- 実運用では `rich-markdown-preview/plan.md` が参考資料内で `annotation-cycle/plan.md` を手動参照している（唯一の実例）

### `/spec` ワークフローの状況

- Step 2（統合分析）で `Glob docs/plans/**/plan.md` を実行して並行開発チェックをしている
- 現在は「変更対象ファイルの重複」のみをチェックし、関連性の提示はしていない

## 調査結果

### 配置場所の比較

| 案 | 位置 | メリット | デメリット |
|----|------|---------|----------|
| 概要の直後に新セクション | `## 概要` と `## 受入条件` の間 | プラン冒頭で文脈把握可能、関連タイプを明示できる | フォーマット変更が必要 |
| 参考資料に統合 | 末尾 | 変更が最小限 | 外部URLと混在、見逃しやすい |
| frontmatter | YAML内 | 機械的パースが容易 | 可読性が低い、YAML配列の記述が煩雑 |

**決定: 概要の直後に `## 関連プラン` セクションを新設**

### 関連プランセクションの設計

```markdown
## 関連プラン

| プラン | 関連 |
|--------|------|
| [annotation-cycle](../annotation-cycle/plan.md) | 本機能の前提。ブラウザレビューの基盤 |
| [live-reload-preview](../live-reload-preview/plan.md) | 同一コンポーネントの後続改善 |
```

設計判断:
- 相対パスを使用（`../` で隣のプランディレクトリを参照）
- 「関連」列に自由記述で関係性を説明（進化元、前提、同一領域 等）
- 型を固定カテゴリにしない。自由記述の方が柔軟で、プラン数が少ないうちは十分

### 自動検出の設計

`/spec` の Step 2 を拡張する:

1. `Glob docs/plans/**/plan.md` で全プラン取得（既存処理）
2. 各プランの frontmatter（title, feature-name）と変更対象ファイルを抽出
3. 以下の基準で関連候補をスコアリング:
   - 変更対象ファイルの重複（既存の並行開発チェックを拡張）
   - feature-name やタイトルのキーワード類似
   - 同一ディレクトリ配下のファイルを対象としている
4. 関連候補をユーザーに提示:
   ```
   以下のプランが関連している可能性があります:
   - annotation-cycle: 変更対象ファイル 2件重複（viewer.html, server.py）
   - live-reload-preview: 同一コンポーネント（annotation-viewer）
   リンクしますか？
   ```
5. ユーザーが選択した関連プランを plan.md の `## 関連プラン` セクションに挿入

### 変更対象

| 変更先 | 内容 |
|--------|------|
| `agents/writer/references/formats/plan.md` | `## 関連プラン` セクション定義を追加（概要の直後、省略可能） |
| `agents/writer/references/examples/plan.md` | 関連プランセクションの具体例を追加 |
| `skills/spec/SKILL.md` | Step 2 の並行開発チェックを拡張し、関連プラン候補の検出・提示を追加 |

## 推奨・結論

- **配置**: `## 概要` の直後に `## 関連プラン` セクションを新設（省略可能セクション）
- **形式**: テーブル形式（プラン名リンク + 関連説明）
- **検出**: `/spec` Step 2 で既存の並行開発チェックを拡張し、関連候補を自動提示
- **リンク**: ユーザーが選択したもののみ挿入（完全自動ではなく半自動）
