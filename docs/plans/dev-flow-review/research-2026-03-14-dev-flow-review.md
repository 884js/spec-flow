# リサーチ: spec-flow 開発フローの総合レビュー

調査日: 2026-03-14
調査タイプ: 複合（コードベース + 設計分析）

## 調査ゴール

spec-flow を使った開発フロー全体を徹底的に調査し、疑問点・改善の余地がある箇所を洗い出す。

## 現状の全体像

### アーキテクチャ

```
(research) -> spec -> build -> check -> done
                ^                |
                |   (NEEDS_FIX)  |
                +----------------+

fix: 独立起動 / list: ナビゲーションハブ
```

- **6スキル**: spec, build, check, fix, research, list
- **4エージェント**: analyzer(opus), writer(opus), verifier(opus), researcher(sonnet)
- **状態管理**: ファイルベース（plan.md / progress.md / result.md）
- **レビュー**: Annotation Cycle（ブラウザベースのコメントUI）

---

## 発見事項・疑問点・改善提案

### 1. list スキルが「案内だけして終わる」問題

**現状**: list でプランを選び、アクションを選ぶと `/spec {feature-name}` を実行してください」と案内メッセージが表示されて **終了する**。ユーザーは別途コマンドを手入力する必要がある。

**疑問**: なぜ直接スキルを呼び出さないのか？Claude Code Plugin の制約でスキルからスキルを呼べないため？

**改善案**: もし技術的に可能なら、list から直接次のスキルを起動できると UX が大幅に改善される。不可能なら、少なくとも案内メッセージを `/spec {feature-name}` のようにコピー&ペーストしやすい形式にする（現状はそうなっている）。

---

### 2. check → spec の NEEDS_FIX ループの自動性

**現状**: check が NEEDS_FIX を出した場合、ユーザーに「`/spec` で仕様を更新してください」と案内し、ユーザーが手動で `/spec` を実行する。spec 側は result.md の存在を検知して更新モードに入る。

**疑問**: このループは設計上は綺麗だが、実際に NEEDS_FIX が出た時の体験はどうか？
- result.md に書かれた不一致箇所が spec 更新モードで十分に活用されているか？
- 更新モードでは analyzer をスキップするが、コード変更後の再分析が必要なケースはないか？

**改善案**:
- check の result.md に「推奨修正アクション」（spec 修正 vs build 修正）を明記し、ユーザーの判断を助ける（現状も一応ある）
- 更新モードでも、変更が大きい場合は analyzer の差分更新モードを活用する選択肢を追加

---

### 3. build の「仕様矛盾検知プロトコル」の実効性

**現状**: build 中に plan.md と実コードの矛盾を検出した場合、3つの選択肢を提示する:
  a) plan.md を修正してから再開
  b) 実装側を仕様に合わせる
  c) このまま進めて後で対応

**疑問**:
- この検知は build スキルのプロンプト指示であり、LLM の判断に完全に依存している。矛盾を見逃すリスクはどの程度あるか？
- 選択肢 c)「このまま進める」を選んだ場合、後で対応すべきことが記録されるか？→ **記録されない**。progress.md にも plan.md にも残らない。

**改善案**:
- 選択肢 c) を選んだ場合、progress.md の「注意事項」セクションに矛盾内容を記録する
- または check で自動的に検出されるはずなので、c) の選択肢自体を削除する

---

### 4. Annotation Cycle の使用率と UX

**現状**: spec 完了後に「ブラウザでレビューする / スキップして次へ」を提示する。server.py は Python 標準ライブラリのみで実装され、plan.md.bak による差分ハイライトも対応済み。

**疑問**:
- 実際にどの程度使われているか？（docs/plans/ 内に comments.json が残っているプランがあるか確認 → 17プラン中で利用実績は不明）
- 30分タイムアウトは適切か？レビューに時間がかかる場合、サーバーが落ちてしまう
- plan.md.bak がクリーンアップされていないケースがある可能性

**改善案**:
- Annotation Cycle の使用を促進するため、初回は自動的にブラウザを開く（スキップは明示的に選択させる）
- plan.md.bak のクリーンアップをサイクル完了時に追加
- comments.json も同様にクリーンアップ（Git にコミットされるべきではない）

---

### 5. progress.md の「現在の状況 / 次にやること」の更新タイミング

**現状**: build がタスク状態を更新する際に「現在の状況」「次にやること」も更新することになっている。

**疑問**:
- コンテキストが長くなった場合（Compaction 後）、LLM がこのフィールドの更新を忘れるリスクは？
- 実際の progress.md を見ると、この情報は build 再開時に有用だが、正確に更新されているか確証がない

**改善案**:
- タスク状態の記号変更（`-` → `→` → `✓`）は機械的なので確実だが、自由テキストの更新は不確実。重要な再開情報はタスク状態から機械的に復元できる設計にする
- または、build スキルのプロンプトでこのフィールドの更新を強調する（現状の記述は控えめ）

---

### 6. researcher エージェントのモデル選択

**現状**: researcher は `model: sonnet` で、他の3エージェント（analyzer, writer, verifier）は全て `model: opus`。

**疑問**:
- researcher が sonnet なのはコスト最適化のためと思われるが、Web 調査の品質に影響しないか？
- 特に「技術比較」「ベストプラクティス調査」のような判断力が必要なタスクでは opus が適切な可能性

**改善案**:
- researcher のタイプ判定を導入: ライブラリのバージョン確認等の事実調査は sonnet、技術比較・アーキテクチャ判断は opus に動的切替
- または現状維持でも問題ないなら、コスト効率の観点で sonnet のまま

---

### 7. fix スキルの debug ドキュメントの活用度

**現状**: fix は feature モードで `debug-{YYYY-MM-DD}-{N}.md` を出力。build は Step 0 で debug ドキュメントを確認し、check も Step 0 で読み込む。

**疑問**:
- standalone モードではドキュメントが保存されない。一度解決した問題が再発した場合、過去の調査結果にアクセスできない
- debug ドキュメントの「plan.md の修正」セクションに記載された修正提案は、誰がいつ plan.md に反映するのか？→ build の Step 0 で「ユーザーに plan.md への反映可否を確認する」となっているが、このフローは忘れられやすい

**改善案**:
- standalone モードでも、プロジェクトルートに `docs/debug/` のような汎用格納先を設けて保存する選択肢を提示
- debug ドキュメントの plan.md 修正提案に「反映済み / 未反映」のステータスを追加

---

### 8. spec の「規模判定」の精度と閾値

**現状**: analyzer の結果からタスク数で規模を判定（小: <3, 中: 3-7, 大: 8+）。大規模の場合は分割提案。

**疑問**:
- タスク数だけで規模を判定するのは粗くないか？1つの L タスクと3つの S タスクでは複雑さが全く違う
- 分割提案を受け入れた場合のフローが不明確。分割後は別の feature-name で `/spec` をやり直すのか？

**改善案**:
- 規模判定に「タスクの見積合計（S=1, M=3, L=5 等のポイント制）」を導入
- 分割時のフローを明確化: 元の plan.md を「Phase 1」に縮小し、Phase 2 以降は別の feature-name で管理する流れを明記

---

### 9. 既存プランの「完了」定義の曖昧さ

**現状**: ステータスは「検証済み」が最終状態だが、PASS / PARTIAL / NEEDS_FIX の区別が list スキルの表示に反映されない。全て「検証済み」として表示される。

**疑問**: PARTIAL で止まったプラン、NEEDS_FIX で spec に戻ったが放置されたプランを区別できるか？

**改善案**:
- list のステータスを拡張: `検証済み(PASS)` / `要修正(NEEDS_FIX)` / `部分合格(PARTIAL)` に分ける
- result.md の最終判定をパースしてステータスに反映

---

### 10. plan.md フォーマットのプロジェクト適応性

**現状**: plan.md フォーマットは「バックエンド変更」「DB変更」「フロントエンド変更」のセクションがあり、Web アプリケーション開発に最適化されている。

**疑問**:
- spec-flow 自体の開発（Markdown ベースのプロンプトエンジニアリング）では、これらのセクションは全て省略される。spec-flow 自体の開発に適したフォーマットになっているか？
- CLI ツール、ライブラリ、インフラ構成など、Web アプリ以外のプロジェクトへの汎用性は？

**改善案**:
- セクション省略ルールが既にあるため、大きな問題ではない。ただし「対象ファイル」セクションがバックエンド/DB/フロントエンドの子セクションとして配置されているため、これらが全て省略されると対象ファイル情報が失われる
- 「変更対象ファイル」を独立したセクションとして設ける、またはタスク表の「対象ファイル」列で代替する（現状もタスク表に含まれている）

---

### 11. エージェントの references/formats の冗長性

**現状**: analyzer, verifier, researcher はそれぞれ `references/formats/output.md` を持ち、writer は `plan.md`, `progress.md`, `result.md` の3つを持つ。

**疑問**:
- verifier の output.md は 31行と非常にシンプル。エージェント定義本体に含めた方がコンテキスト効率が良いのでは？
- researcher の output.md も 45行。同様に本体に含めることも検討

**改善案**:
- ルール（agent-authoring.md）で「フォーマットは references/ に配置」と定められているため、一貫性のためにこのまま維持するのが妥当
- ただし、エージェントが毎回 Read するオーバーヘッドがある。頻繁に呼ばれるエージェントは本体に埋め込む方がトークン効率が良い可能性

---

### 12. .gitignore の未整備

**現状**: Annotation Cycle で生成される一時ファイル（`comments.json`, `plan.md.bak`）が `.gitignore` に含まれていない可能性がある。

**疑問**: これらのファイルがコミットされるとノイズになる。

**改善案**: `.gitignore` に以下を追加:
```
docs/plans/*/comments.json
docs/plans/*/plan.md.bak
```

---

### 13. build → PR 作成フローの欠落

**現状**: build スキルの Step 4 は「ビルド確認」で終わり、PR 作成の手順が明記されていない。description には「PR creation」と書いてあるが、SKILL.md 本文には PR 作成ステップがない。

**疑問**: build 完了後の PR 作成はどうやっているのか？`/commit` スキルを使っている？それとも手動？

**改善案**:
- build の Step 4 に PR 作成フローを追加する
- または、build は実装のみに集中し、PR 作成は別スキル（commit）に任せることを明記する

---

### 14. multi-pr モードの実際の運用実績

**現状**: spec で大規模機能を分割し、progress.md に PR 列を追加して multi-pr モードで管理する設計。

**疑問**:
- 既存の17プランを確認すると、全て `mode: single` で運用されている。multi-pr モードの実績がない
- 実際に大規模機能を分割する場面が発生した時、このフローは期待通りに動くか検証されていない

**改善案**:
- multi-pr モードは未検証のため、実際のユースケースが発生してから改善する方針でよい
- ただし、ドキュメント上「サポート」と記載されていることは注意。未検証であることを明記するか、小規模な検証を行う

---

### 15. エージェントの「自己完結パターン」と実際の依存

**現状**: ルール上、エージェントは自身の `references/` 配下のみを参照する。

**疑問**:
- writer は `plan.md` のフォーマット定義を自身の references に持っているが、analyzer の出力フォーマットと plan.md のフォーマットに暗黙の依存がある（analyzer が報告した情報を writer が plan.md のセクションにマッピングする必要がある）
- この暗黙の依存はスキル側のプロンプトで吸収されているが、analyzer の出力フォーマットを変更した場合にスキルのプロンプトも更新が必要

**改善案**:
- 現状のスキルがオーケストレーション層として仲介しているのは適切な設計。ただし、この依存関係を明文化しておくと保守性が上がる
- 例えば skills/README.md にエージェント間のデータフロー図を追加

---

## 全体評価

### 強み
1. **ファイルベース状態管理** — state.json を使わず、plan.md / progress.md / result.md の存在で状態を判定する設計は堅牢でデバッグしやすい
2. **スキル/エージェント分離** — オーケストレーション（スキル）と実処理（エージェント）の分離が明確
3. **NEEDS_FIX ループ** — check → spec への自動フィードバックは仕様駆動開発の核心を実現している
4. **Annotation Cycle** — ブラウザベースのレビューは、ターミナルだけでは難しい仕様レビューを可能にしている
5. **fix の「原因特定まで修正禁止」原則** — 推測的修正を防ぐ設計は品質向上に直結
6. **自己検証の組み込み** — writer の生成中検証、analyzer の信頼度ラベルは品質担保に貢献

### 改善優先度（提案）

| 優先度 | 項目 | 理由 |
|--------|------|------|
| 高 | #13 build の PR 作成フロー追記 | description と実装の乖離。ユーザーが迷う |
| 高 | #12 .gitignore 整備 | 一時ファイルのコミット防止 |
| 高 | #9 list の NEEDS_FIX ステータス表示 | 放置プランの可視化 |
| 中 | #3 仕様矛盾の「後で対応」記録 | 情報の消失防止 |
| 中 | #4 Annotation Cycle の一時ファイル管理 | plan.md.bak / comments.json のクリーンアップ |
| 中 | #8 規模判定の精緻化 | 分割判断の精度向上 |
| 低 | #1 list からの直接スキル起動 | UX 改善だが Plugin の制約次第 |
| 低 | #6 researcher のモデル動的切替 | コスト vs 品質のトレードオフ |
| 低 | #14 multi-pr モードの検証 | 利用実績がないため後回し |

## 次のステップ

- 優先度「高」の項目について `/spec` で仕様を作成し、改善を実装する
- Annotation Cycle の利用実績データを収集し、UX 改善の方向性を決める
- multi-pr モードは実際のユースケースが発生してから検証する
