# リサーチ: /build 実行前のタスクプレビュー・選択機能

調査日: 2026-03-12
調査タイプ: コードベース

## 調査ゴール

`/build` 実行前に plan.md のタスク一覧をユーザーに提示し、実装するタスクを取捨選択できるようにする最適な差し込み箇所とUXパターンを明らかにする。

## 現状

### build スキルのワークフロー

| Step | 内容 | 概要 |
|------|------|------|
| 0 | 読み込み + 再開検知 | plan.md / progress.md 読み込み、再開判定 |
| 1 | feature ブランチ作成 | AskUserQuestion でベースブランチ確認 |
| 2 | タスク順の実装 | 依存関係順に全タスクを自動実行 |
| 3 | ビルド確認 | コマンド実行 + 手動検証確認 |
| 4 | PR 作成 | push + `gh pr create` |

**問題**: Step 0 → Step 2 で**全タスクが自動的に実行される**。ユーザーがタスクを選ぶ機会がない。

### plan.md のタスク表構造

```
| # | タスク | 対象ファイル | 見積 | 依存 |
```

### progress.md の状態管理

- 状態: `-`（未着手）→ `→`（実施中）→ `✓`（完了）
- multi-pr 時は PR 列で対象タスクを絞る機構あり（唯一のスコープ制限）

## 調査結果

### 差し込み箇所の比較

| 箇所 | 位置 | メリット | デメリット |
|------|------|----------|-----------|
| **Step 0 と Step 1 の間（推奨）** | SKILL.md L99 後 | 全情報が揃い副作用なし。新規・再開両方に対応 | なし |
| Step 0 再開検知の分岐内 | SKILL.md L94-97 | 影響範囲が小さい | 再開時に選択変更不可 |
| Step 2 PR スコープ拡張 | SKILL.md L121-123 | 既存機構の拡張 | single モード別途対応が必要 |

### AskUserQuestion の既存パターン

spec-flow 内の AskUserQuestion 利用パターンを分析した結果、タスク選択には **2段階方式** が最適:

1. **第1段階**: 「全タスクを実行」/「タスクを選択する」の2択
2. **第2段階**: 選択時のみ個別タスクを `multiSelect: true` で列挙

理由:
- タスク数が少ない場合（2-3個）に毎回選択を強いるのはUXが悪い
- 既存の「件数に応じた分岐型」パターン（fix スキル）と整合性が高い

### 依存関係チェックの必要性

タスク間に依存関係がある場合（`依存` 列）、依存先を外して依存元を選ぶと実装が破綻する。

対策案:
- 依存先タスクが未選択の場合、**警告を表示して自動的に依存先も選択に含める**
- または依存関係グラフを可視化してから選択させる（過剰かもしれない）

### progress.md への反映

選択されなかったタスクの扱い:
- `skip`（スキップ）状態を新設するか、単に `-` のまま残すか
- 推奨: `-` のまま残す（再開時に再度選択可能にするため）

## 推奨・結論

**Step 0 と Step 1 の間に「Step 0.5: タスクプレビュー + 選択」を追加する。**

```
Step 0.5: タスクプレビュー + 選択
  1. plan.md のタスク表をフォーマット付きで提示
  2. AskUserQuestion（第1段階）:
     - 「全タスクを実行する」（Recommended）
     - 「実装するタスクを選択する」
  3. 選択モードの場合、AskUserQuestion（第2段階, multiSelect: true）:
     - 各タスクを選択肢として列挙（タスク名 + 見積 + 対象ファイル）
  4. 依存関係チェック → 不足があれば警告付きで自動補完
  5. 選択結果を以降の Step で処理対象として引き継ぐ
```
