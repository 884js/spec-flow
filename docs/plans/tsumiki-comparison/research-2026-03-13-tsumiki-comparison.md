# リサーチ: tsumiki 記事から spec-flow に参考にできる要素の調査

調査日: 2026-03-13
調査タイプ: 複合（外部技術 + コードベース比較）

## 調査ゴール
Zenn 記事「Claude Codeでの仕様駆動開発：tsumiki活用ガイド」で紹介されている手法と spec-flow を比較し、取り入れられる改善点を特定する。

## 記事の要点

記事の基本思想は「思考は人間、実装はAI」。事前に仕様精度を高めることで一発実装を目指す。

### tsumiki のワークフロー

| フェーズ | 担当 | 内容 |
|---------|------|------|
| 1. 要件定義 | 人間 + AI 対話 | AIが徹底的に質問し、見落とし観点を自動抽出 |
| 2. 設計 | 人間 + AI | requirements.md / design.md を生成 |
| 3. タスク化 | AI | 実装可能な単位へ分割 |
| 4. 実装 | AI 自動 | TDD サイクル（Red → Green → Refactor → Verify） |

### tsumiki の特徴的な仕組み

1. **信号機システム**: 仕様の信頼度を3段階（青=確実/黄=要確認/赤=要議論）で可視化
2. **TDD 実装サイクル**: テスト先行の Red → Green → Refactor → Verify
3. **徹底的な質問機能**: 要件定義で人間の見落としを自動抽出
4. **ファイル分離**: requirements.md と design.md を分離

## 現状（spec-flow との比較）

| 観点 | spec-flow | tsumiki | 差分 |
|------|-----------|---------|------|
| 仕様書の構成 | plan.md 1ファイルに統合 | requirements.md + design.md 分離 | spec-flow の方がシンプル |
| 要件ヒアリング | AskUserQuestion で1往復→規模判定 | 徹底的な質問で見落とし抽出 | tsumiki の方が深い |
| 仕様の信頼度可視化 | なし | 信号機システム（青/黄/赤） | **spec-flow にない機能** |
| 実装サイクル | タスク順実装、サイズ別進め方 | TDD（Red→Green→Refactor→Verify） | **アプローチが異なる** |
| 検証 | /check で仕様と実装を突合 | TDD の Verify ステップで自動検証 | spec-flow は事後検証、tsumiki はインライン検証 |
| フェーズ遷移 | ファイルベース推論 + Hooks | 不明（記事に記載なし） | spec-flow の方が明確 |
| ブラウザレビュー | Annotation Cycle（server + viewer） | なし | **spec-flow 独自** |
| 技術調査 | /research スキル | なし（記事に記載なし） | **spec-flow 独自** |
| バグ修正 | /fix スキル | なし（記事に記載なし） | **spec-flow 独自** |
| プラン管理 | /list でハブナビゲーション | なし | **spec-flow 独自** |

## spec-flow に取り入れられる可能性のある要素

### 1. 信号機システム（仕様の信頼度可視化）— 推奨度: ★★★

**概要**: plan.md 内の各定義に信頼度ラベル（青=確実/黄=要確認/赤=要議論）を付与する。

**spec-flow への適用案**:
- spec スキルの plan.md 生成時に、analyzer の情報から判定:
  - 🔵 既存コードや明示的要件に基づく定義
  - 🟡 合理的な推測だが確認が必要な定義
  - 🔴 根拠のない推測・要議論項目
- Annotation Cycle でレビュー時に黄・赤を重点的に確認できる

**メリット**: 仕様の「怪しい部分」が一目で分かり、レビュー効率が上がる。記事著者は「赤信号と黄信号を潰せばほぼ一発で実装できた」と評価。

**懸念**: plan.md のフォーマットが複雑になる。writer エージェントの出力フォーマット変更が必要。

### 2. 要件ヒアリングの深化 — 推奨度: ★★☆

**概要**: 要件定義フェーズで、AIが能動的に「見落としがちな観点」を質問する。

**spec-flow への適用案**:
- spec スキルの Step 1（要件ヒアリング）で、ユーザー入力に対して自動的に:
  - エッジケースの質問
  - 非機能要件（パフォーマンス・エラーハンドリング）の確認
  - 既存機能との整合性確認
- 現在の「1往復で明確になったら次へ」から「重要な観点が網羅されるまでヒアリング」に変更

**メリット**: 仕様の精度が上がり、build 中の仕様矛盾が減る。

**懸念**: ヒアリングが長くなりすぎると UX が悪化する。バランスが重要。

### 3. TDD 実装サイクルの導入 — 推奨度: ★★☆

**概要**: build フェーズでテスト先行の実装サイクルを採用する。

**spec-flow への適用案**:
- build スキルのタスク実装ステップに TDD モードを追加:
  1. Red: タスクの受入条件からテストを先に書く
  2. Green: テストが通る最小実装
  3. Refactor: コード品質の改善
  4. Verify: テスト実行で確認

**メリット**: 実装品質の向上、/check での検証がより確実になる。

**懸念**: 全プロジェクトでテスト環境が整っているとは限らない。plan.md にテスト方針の記載が必要。オプション化が現実的。

### 4. CLAUDE.md / rules 整備の重要性の強調 — 推奨度: ★☆☆

**概要**: 記事は「実装が詰まる原因は仕様でなくプロジェクトルール不備」と断言。

**spec-flow への適用案**:
- spec スキルの初回実行時に CLAUDE.md と .claude/rules/ の存在をチェックし、不足していれば整備を提案する

**メリット**: 実装品質のベースラインが上がる。

**懸念**: spec-flow の責務範囲を超える可能性がある。

## spec-flow が既に優位な点

1. **ファイルベースフェーズ推論**: state.json を使わない設計で状態不整合が起きにくい
2. **Annotation Cycle**: ブラウザでのビジュアルレビュー + コメント反映は tsumiki にない
3. **Hooks による自動遷移提案**: phase-detector.sh でフェーズ完了を自動検知
4. **research / fix / list スキル**: 調査・バグ修正・ナビゲーションまでカバー
5. **スキル-エージェント分離**: 明確なアーキテクチャパターン

## 推奨・結論

最も取り入れる価値が高いのは **信号機システム**。仕様の信頼度を可視化することで、Annotation Cycle でのレビュー効率が大幅に向上する可能性がある。spec-flow の既存の強み（Annotation Cycle + writer エージェント）との親和性も高い。

要件ヒアリングの深化と TDD サイクルもメリットはあるが、UX とのバランスやオプション化の設計が必要。段階的に検討するのが現実的。

## 次のステップ

- 信号機システムの導入を `/spec` で検討する場合 → plan.md フォーマットへの組み込み方を設計
- TDD モードを `/build` に追加する場合 → オプション設計と plan.md のテスト方針セクション追加を検討
- ヒアリング深化を試す場合 → spec スキルの Step 1 を改修

## spec-flow 流アレンジ案（思想の取り込み）

直接コピーではなく、spec-flow の既存アーキテクチャに自然に統合する形で思想を取り込む。

### A. 「不確実性の可視化」→ Confidence Annotation

**tsumiki の思想**: 仕様内の各定義に信頼度ラベルを付け、曖昧なまま実装に突入しない。

**spec-flow 流のアレンジ**: plan.md 内に「確認事項」セクションを追加し、analyzer が検出した推測・仮定を明示化する。

#### 具体案: plan.md に「確認事項（Assumptions）」セクションを追加

```markdown
### 確認事項

| # | 項目 | 根拠 | ステータス |
|---|------|------|----------|
| A-1 | ユーザーテーブルに email カラムが unique 制約あり | 既存 schema から確認済み | ✅ 確認済み |
| A-2 | 通知はメールのみ（Slack 不要） | ヒアリングで未確認 | ⚠️ 要確認 |
| A-3 | バッチ処理の実行頻度は1時間ごと | 推測（類似機能から類推） | ❓ 要議論 |
```

**ポイント**:
- 信号機の「色」ではなく、✅/⚠️/❓ のステータスで表現（plan.md の他セクションと統一感）
- analyzer が「既存コード影響分析」で検出した情報 → ✅ 確認済み
- ヒアリングで触れなかった観点 → ⚠️ 要確認
- 根拠なく仮定した設計判断 → ❓ 要議論

**Annotation Cycle との連携**:
- ブラウザレビュー時に ⚠️/❓ の項目が目立つようにハイライト可能
- レビューで確認したら ✅ に更新 → 全項目 ✅ になったら「仕様確定」

**実装の影響範囲**:
- `agents/writer/references/formats/plan.md` にセクション定義を追加
- `skills/spec/SKILL.md` の Step 2（analyzer）で「推測した項目」を収集する指示を追加
- `skills/spec/SKILL.md` の Step 3（方向性確認）で ⚠️/❓ 項目を提示

### B. 「ヒアリングの深化」→ analyzer 駆動の観点チェック

**tsumiki の思想**: AIが能動的に見落としがちな観点を質問し、要件の穴を埋める。

**spec-flow 流のアレンジ**: 現在の「1-2往復で次へ」は変えず、analyzer の結果を使って **Step 3（方向性確認）で** 見落とし観点を提示する。

#### 具体案: Step 3 に「観点チェック」を追加

現在の Step 3 は「方向性サマリ → 確認」の2ステップ。ここに analyzer が検出した「仕様で触れていない観点」を追加する。

```
Step 3-a: 方向性サマリ（現行通り）
Step 3-b: 観点チェック（新規）
Step 3-c: 確認（現行通り）
```

**Step 3-b の内容**:
analyzer の統合レポートから、ユーザーの要件で触れていないが影響しそうな観点を抽出:

```
以下の観点がまだ決まっていません:
- エラー時のユーザーへの通知方法（既存コードでは toast を使用）
- 既存の〇〇機能との権限の整合性
- データ量が増えた場合のページネーション

確認が必要ですか？それとも既存パターンに従いますか？
```

**ポイント**:
- ヒアリング（Step 1）は軽く保つ。ユーザーの「やりたいこと」を素早く把握するフェーズ
- 深い質問は analyzer の結果が出た **後に** 行う（コードベースの文脈を踏まえた質問ができる）
- 「既存パターンに従う」を選べるので、毎回深い議論を強制しない

**実装の影響範囲**:
- `agents/analyzer/analyzer.md` に「仕様で未言及の観点」を出力する指示を追加
- `skills/spec/SKILL.md` の Step 3 に 3-b を追加
- analyzer の output フォーマットに「未言及観点」セクションを追加

### A + B の相乗効果

```
Step 1: ヒアリング（軽く）
Step 2: analyzer 実行 → 推測項目 + 未言及観点を収集
Step 3: 方向性確認 + 観点チェック（⚠️/❓ を提示）
Step 4: plan.md 生成（確認事項セクション付き）
Step 4-c: Annotation Cycle（⚠️/❓ をレビューで潰す）
→ 全項目 ✅ = 仕様確定 = 高精度な実装へ
```

既存フローに **2つのタッチポイント**（Step 3-b と plan.md の確認事項セクション）を追加するだけで、「不確実性を明示して潰す」思想を取り込める。
