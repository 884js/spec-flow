# 不具合調査: PreToolUse で Skill ツールのスキル名が取得できない

調査日: 2026-03-13

## 症状
パターン: 期待 vs 実際
- 期待: PreToolUse (Skill) 発火時に `tool_input.skill` からスキル名を取得し、状態ファイルに保存
- 実際: `SKILL_NAME` が常に空。debug ログで全 PreToolUse イベントの tool_name を集計した結果、Bash/Write/Edit のみ。Skill ツールの PreToolUse イベントは 0 件

## 原因
Claude Code は **Skill ツール呼び出し時に PreToolUse フックを発火しない**。

debug ログ（`/tmp/phase-detector-debug.log`）の全 tool_name を集計:
```
3 "tool_name":"Write"
3 "tool_name":"Bash"
1 "tool_name":"Edit"
```
セッション開始時に `/check`（= Skill ツール）を実行したが、`tool_name:"Skill"` は記録されなかった。

これは Claude Code の仕様と推測される。フック対象の PreToolUse スキーマに Skill ツールが記載されていないことと一致する（公式ドキュメントでは Bash, Write, Edit, Read, Glob, Grep, WebFetch, WebSearch, Agent の 9 ツールのみ記載）。

## 修正方針
### コード修正
PreToolUse (Skill) による状態保存は不可能。代替アプローチ:

| 候補 | 方式 | メリット | デメリット |
|------|------|---------|----------|
| A | UserPromptSubmit でユーザー入力から `/skill-name` パターンを検出 | 追加依存なし | スキル名のパース精度が不安定 |
| B | PostToolUse で spec-flow 関連のファイル操作（plan.md, progress.md 等）からスキルを推測 | 確実にファイルを検知可能 | スキル名は推測止まり |
| C | スキル追跡機能を廃止し、PostToolUse のフェーズ検知のみに絞る | 最もシンプル。PostToolUse のフェーズ検知は正常動作確認済み | UserPromptSubmit のコンテキスト注入が失われる |

### plan.md の修正
- データフロー図の「PreToolUse (Skill) → スキル名を状態ファイルに保存」を修正
- 設計判断テーブルに Skill ツールの PreToolUse 非対応を追記
- 自動テスト #5（PreToolUse でスキル名保存）を削除または修正

### デバッグログの除去
- `hooks/phase-detector.sh` L26-27: `[debug]` + `[debug-input]` ログ出力
- `hooks/phase-detector.sh` L32: PreToolUse の SKILL_NAME ログ
- `hooks/phase-detector.sh` L107: UserPromptSubmit の一時ログ
- `hooks/phase-detector.sh` L148: stdout output ログ
